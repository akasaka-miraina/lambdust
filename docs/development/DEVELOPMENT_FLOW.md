# 開発フロー

## 🔄 開発フローの遵守

プロジェクトではpre-commitフックを使用してコード品質を自動チェックしています：

- **Clippy**: コードの静的解析とリント
- **Tests**: 全テストの実行とパス確認  
- **Documentation**: ドキュメントビルドの成功確認
- **Formatting**: コードフォーマットの確認（警告のみ）

コミット前に自動的にこれらのチェックが実行され、すべてグリーンシグナルであることが確認されます。

## 🎯 品質管理方針

### 基本原則：「隠す」ではなく「直す」

品質管理セクションの指摘および開発コードの保守性と品質保持の観点から、以下の方針を徹底します：

#### 1. **Linter警告への対処**
- ❌ **禁止**: `#[allow(...)]`による警告抑止の使用
- ✅ **推奨**: 根本的なコード改善による警告解決
- 🔧 **例外**: 技術的制約により回避不可能な場合のみ最小限使用

#### 2. **コンパイラ警告への対処**
- ❌ **禁止**: 警告の隠蔽や一時的回避策
- ✅ **推奨**: 
  - 未使用コード→適切な活用メソッド実装
  - 型警告→正確な型注釈とエラーハンドリング
  - デッドコード→リファクタリングまたは適切な削除

#### 3. **Rustfmt警告への対処**
- ❌ **禁止**: `#[rustfmt::skip]`による整形回避
- ✅ **推奨**: 
  - 自動整形に適合するコード構造
  - 可読性を損なわない適切なコード分割
  - 標準的なRustコーディングスタイルの遵守

### コーディング原則遵守

#### DRY原則（Don't Repeat Yourself）

- 重複コードの積極的統合
- 共通ロジックの適切な抽象化
- 再利用可能なモジュール設計

#### 単一責務原則（Single Responsibility Principle）

- 各モジュール・関数の責務明確化
- 適切な関心の分離
- 凝集度の高い設計

#### KISS原則（Keep It Simple, Stupid）

- 不要な複雑性の排除
- シンプルで明快なコード構造

#### モジュール化原則

- 1struct/enum定義とImplで1.rsファイルとする

### 開発に関する**重要禁止**事項

- allowによるClippy警告無視の禁止
- 3段以上のネスト禁止
- 100行以上のメソッド禁止
- 1,000行以上のrsファイル作成禁止

### 品質チェックプロセス

注意：0.3.0より施行．

1. **開発時**: `cargo clippy --all-targets --all-features -- -A clippy::all -D warning`で警告ゼロ確認
2. **コミット前**: pre-commitフックによる自動品質検証
3. **PR作成時**: 全テスト通過と警告解決確認
4. **継続改善**: 技術的負債の計画的解消

### 品質目標

- **Warning Free**: プロダクションコードでの警告完全ゼロ化
- **Test Coverage**: 高い品質のテストカバレッジ維持
- **Maintainability**: 保守性の高いコード構造
- **Performance**: 最適化とコード品質の両立
- **value immutability**: 安易なmut修飾に頼らず，ライフタイム付きBoxなどを活用する
- **value borrow**: 安易なcloneに頼らず，参照や借用を重要視する
- **Perfect abnormal test**: 周到な異常テスト

この方針により、短期的な開発効率よりも長期的な保守性と品質を重視し、持続可能な開発体制を確立します。

### 基本的な作業手順

1. **Issue作成**: GitHubでIssueを作成し、作業内容を明確化
2. **ブランチ作成**: mainブランチからfeatureブランチをfork
3. **設計・実装**: 機能の設計と実装を行う
4. **テスト・品質チェック**: `make dev-check`でlint・test・フォーマット確認
5. **コミット**: pre-commitフックによる自動品質チェック後コミット
6. **進捗追記**: CLAUDE.mdに完了した機能・ステータスを追記
7. **Pull Request**: GitHub CopilotのレビューコメントあるPRを作成
8. **レビュー・マージ**: コードレビュー後、mainブランチにマージ

### 🔄 重要な開発フロー原則

- **必須**: 各機能完了後に必ずCLAUDE.mdに進捗を記録
- **必須**: テスト追加とpre-commitフック通過
- **推奨**: 大きな機能完了時にコミット・進捗追記のセット実行
